AWS IAM Basics: Users, Groups, and Permissions

AWS Identity L1 30–45 mins

This lab introduces AWS Identity and Access Management (IAM) and teaches how to create users, groups, and permissions safely using best practices.

Start Lab →

What You’ll Build

  • An IAM user for day-to-day access
  • An IAM group with attached permissions
  • A least-privilege access model
  • An understanding of why the root user should not be used

Prerequisites

  • An AWS account
  • Access to the AWS Management Console
  • No prior IAM experience required

Lab Steps

Step 1 — Understand IAM and the Root User

Objective

Understand what IAM is and why the root user should not be used for everyday work.

Simple Explanation

AWS IAM controls who can do what in your AWS account.

The root user:

  • Has full control of the account
  • Cannot be restricted
  • Should only be used for initial setup

What You Should Do

  • Log in to AWS as the root user
  • Notice the warning banners
  • Do not use the root user for daily work

Step 2 — Create an IAM Group

Objective

Create a group to manage permissions centrally.

  1. Open the IAM console
  2. Click User groups
  3. Click Create group
  4. Name the group Developers
  5. Attach the ReadOnlyAccess policy
  6. Create the group

Why This Matters

Groups prevent permission sprawl and reduce mistakes by managing access in one place.

Step 3 — Create an IAM User and Add to the Group

Objective

Create a non-root IAM user for daily work.

  1. Go to Users in IAM
  2. Click Create user
  3. Name the user dev-user
  4. Enable AWS Management Console access
  5. Add the user to the Developers group
  6. Complete user creation

Verification

  • Sign in as dev-user
  • Confirm access is read-only

Step 4 — Attach Least-Privilege Policies

Objective

Replace overly broad permissions with a safer, least-privilege model.

Simple Explanation

Least privilege means giving users only the permissions they need—nothing more.

  1. Open the IAM console
  2. Go to User groups → Developers
  3. Attach a more specific policy (for example, AmazonS3ReadOnlyAccess)
  4. Remove policies that are not required

Why This Matters

Least privilege limits blast radius if an account is compromised.

Step 5 — Enable Multi-Factor Authentication (MFA)

Objective

Protect IAM users with a second authentication factor.

Simple Explanation

MFA requires something you know (password) and something you have (code).

  1. Open IAM
  2. Select Users → dev-user
  3. Open Security credentials
  4. Assign a virtual MFA device
  5. Complete MFA setup

Verification

  • Sign out
  • Sign back in
  • Confirm MFA is required

Why This Matters

MFA significantly reduces the risk of account takeover.

Step 6 — Audit IAM Activity Using CloudTrail

Objective

Learn how to see and prove what actions IAM users perform.

Simple Explanation

Security teams must answer: “Who did what, and when?”

  1. Open the AWS CloudTrail console
  2. Go to Event history
  3. Filter by iam.amazonaws.com
  4. Review recent IAM actions

What This Proves

  • IAM activity is logged
  • Changes are auditable
  • Incidents can be investigated

End of Step 6 — Lab Complete

  • IAM users are created safely
  • Permissions follow least privilege
  • MFA protects identities
  • Activity is auditable and reviewable
