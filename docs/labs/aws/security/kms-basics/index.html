AWS KMS Basics: Keys, Encryption, and Trust

AWS Security L1 30–45 mins

This lab introduces AWS KMS and explains how encryption keys act as trust anchors for protecting data across AWS services.

Start Lab →

What You’ll Build

  • An AWS customer-managed KMS key (CMK)
  • Clear rules for who can encrypt and decrypt
  • Encryption using a CMK with an AWS service
  • Audit evidence of key usage in CloudTrail

Prerequisites

  • An AWS account
  • Access to the AWS Management Console
  • No prior KMS experience required

Lab Steps

Step 1 — Understand What KMS Is (Foundation)

Objective

Understand what AWS KMS does and why keys matter.

Simple Explanation

Encryption locks data. Keys decide who gets the lock.
AWS KMS is the service that creates, stores, and controls those keys.

There are two common types:

  • AWS-managed keys (easy, limited control)
  • Customer-managed keys (CMKs) (you control permissions and audit)

What You’ve Learned

  • Keys are trust boundaries
  • Controlling keys = controlling access

Step 2 — Create a Customer-Managed KMS Key

Objective

Create a CMK you control.

Actions

  1. Open AWS KMS
  2. Click Create key
  3. Choose Symmetric → Encrypt and decrypt
  4. Name the key: stc-kms-basics-key
  5. Leave default admins (your account)
  6. Allow your account to use the key
  7. Create the key

What Changed

  • You now own a key
  • Permissions to use it are explicit and auditable

Step 3 — Use the Key and Verify Auditability

Objective

Prove the key works and that usage is logged.

Actions

  1. Open S3 (or another supported service)
  2. Enable SSE-KMS using stc-kms-basics-key
  3. Upload a test file
  4. Open CloudTrail
  5. Filter for KMS events

Verify

  • You see Encrypt / Decrypt events
  • The key ID matches your CMK

Why This Matters

  • You can prove who used the key
  • This is required for compliance and forensics
