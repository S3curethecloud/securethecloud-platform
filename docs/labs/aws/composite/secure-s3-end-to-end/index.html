AWS Composite L2 60–90 mins

This lab combines multiple AWS security controls to build a fully secured S3 bucket with enforced encryption, controlled access, and audit visibility.

What You’ll Build

  • An S3 bucket protected end-to-end
  • IAM permissions enforcing least privilege
  • KMS-controlled encryption keys
  • Bucket policies preventing misconfiguration
  • Audit trails proving security enforcement

Zero Trust Mapping — Identity, Data Protection, Observability

This composite lab intentionally maps to multiple Zero Trust pillars. Rather than treating controls in isolation, it demonstrates how trust is enforced continuously across identity, data, and audit layers.

  • Identity: IAM roles and policies define exactly who is allowed to access S3 and use encryption keys.
  • Data Protection: KMS-backed encryption and S3 bucket policies ensure data remains protected even if access is attempted.
  • Observability: CloudTrail provides immutable audit logs proving who acted, what was accessed, and when.

Together, these controls enforce Zero Trust by design: trust is never assumed, access is explicitly granted, and every action is observable and provable.

Lab Steps

Step 1 — Design the Security Model (Before Building)

Objective

Define how identity, data, keys, and logs work together.

  • IAM controls who can act
  • KMS controls who can encrypt and decrypt
  • S3 policies control what is allowed
  • CloudTrail proves what happened

Security must be designed before resources exist.

Step 2 — Create IAM Role and Permissions

  1. Create an IAM role or user for S3 access
  2. Grant s3:PutObject and s3:GetObject
  3. Grant permission to use a specific KMS key
  4. Do not grant admin permissions

Least privilege is enforced at the identity layer.

Step 3 — Create KMS Key and Secure S3 Bucket

  1. Create a customer-managed KMS key
  2. Create an S3 bucket
  3. Enable SSE-KMS using your CMK
  4. Block all public access

Data is protected even if access is attempted.

Step 4 — Enforce Security with Bucket Policies

  • Deny unencrypted uploads
  • Deny public access explicitly
  • Bind encryption to your CMK

Outcome: Misconfigurations fail safely.

Step 5 — Validate and Attempt Violations

  • Upload without encryption → fails
  • Attempt public access → fails
  • Access with correct IAM role → succeeds

Outcome: Controls behave as designed.

Step 6 — Audit with CloudTrail

  • Review CloudTrail events
  • Confirm who accessed S3
  • Confirm which KMS key was used
  • Confirm when actions occurred

Outcome: Security is provable.
