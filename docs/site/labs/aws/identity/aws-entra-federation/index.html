AWS ↔ Entra ID Workforce Identity Federation

AWS Entra ID Identity L4 60–90 mins

This lab guides you through implementing production-grade workforce identity federation between AWS IAM Identity Center and Microsoft Entra ID using SAML for authentication and SCIM for authorization.

Start Lab →

What You’ll Build

  • AWS IAM Identity Center integrated with Microsoft Entra ID
  • SAML-based authentication between AWS and Entra
  • SCIM user and group provisioning
  • Role-based access across cloud environments

Prerequisites

  • An AWS account with IAM Identity Center enabled
  • A Microsoft Entra ID tenant with admin permissions
  • Basic understanding of IAM and SAML concepts

Lab Steps

Step 1 — Prepare Identity Systems (Foundation)

Objective

Ensure both identity systems (AWS and Microsoft Entra ID) are ready to participate in federation.

What You Are Establishing

  • Entra ID as the Identity Provider (IdP)
  • AWS IAM Identity Center as the Service Provider (SP)

Actions

  • Confirm AWS IAM Identity Center is enabled
  • Confirm Entra ID tenant admin access
  • Identify workforce users and groups

Why This Matters

Federation failures often stem from misaligned foundations. This step ensures readiness before trust is established.

Step 2 — Create the Federation Trust (SAML)

Objective

Establish SAML-based authentication trust between Entra ID and AWS.

Actions

  • Configure SAML provider in AWS IAM Identity Center
  • Create Entra Enterprise Application
  • Upload AWS metadata and configure claims

Why This Matters

This defines the cryptographic trust boundary. Errors here break all access.

Step 3 — Enable SCIM Provisioning (Authorization)

Objective

Automate user and group lifecycle management from Entra ID into AWS.

Actions

  • Generate SCIM endpoint and token in AWS
  • Configure SCIM provisioning in Entra ID
  • Enable user and group sync

End of Step 3 — What You Have Achieved

  • Authentication flows from Entra ID to AWS
  • Authorization is automated
  • Manual IAM users eliminated

Step 4 — Configure Permission Sets & Group Mapping

Objective

Define what federated users are allowed to do in AWS.

Actions

  • Create permission sets (ReadOnly, PowerUser, etc.)
  • Attach IAM policies
  • Assign permission sets to accounts
  • Map Entra ID groups to permission sets

Why This Matters

This establishes RBAC. Access is granted only through group membership.

Step 5 — Validate Federated Access & Sessions

Objective

Confirm authentication and authorization work end-to-end.

Actions

  • Sign in via Entra My Apps
  • Launch AWS application
  • Select account and role
  • Verify permissions

Why This Matters

Confirms the federation pipeline is functional.

Step 6 — Audit, Monitor, and Harden the Federation

Objective

Ensure federation remains secure and auditable.

Actions

  • Review CloudTrail federated sign-ins
  • Test SCIM deprovisioning
  • Apply Entra conditional access policies

End of Step 6 — Lab Complete

  • Authentication centralized
  • Authorization group-driven
  • Access auditable and enforced
  • AWS identity fully modernized
